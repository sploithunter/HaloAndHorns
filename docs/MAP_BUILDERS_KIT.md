# Map Builder's Kit — Pet Realm (Halo & Horns)

**Audience:** a map builder (contracted or in-house) producing the production world.
**Promise:** you build geometry and place invisible markers; you never touch code. Every
gameplay system binds to the map at boot through tags, attributes, and a few reserved
names. If a required hook is missing, the game fails LOUDLY at boot with the exact path —
nothing silently half-works.

Companion references: `docs/MAP_MARKER_REFERENCE.md` (tag/attribute schemas),
`docs/wiki/MAP_INTEGRATION_CONTRACT.md` (how binding works),
`docs/AUTHORED_MAP_WORKFLOW.md` (Studio workflow), `configs/areas.lua` (zone ids),
`configs/layers.lua` (layer ladder, portals, skyboxes).

---

## 1. What this game is (the 60-second brief)

Pet Realm is a **heaven-and-hell directional pet game** with City-of-Heroes combat bones.
Players mine crystals, hatch and grow a squad of pets, fight alongside them with a power
kit, and choose a *direction*: ascend through five Heaven layers or descend through five
Hell layers. Pets are the stars — the camera lives at player scale, and the world is the
stage for squads of up to 10 pets per player plus enemies.

The realm duality is mechanical, not just cosmetic:

- **Heaven = coexistence.** Heaven-aligned enemies only attack hell-aligned targets, so
  heaven reads as radiant, peaceful, farm-friendly — danger is *chosen*.
- **Hell = free-for-all.** Hell attacks everything. Hell reads as hostile, ember-lit, and
  (deeper down) genuinely DARK — by design, the counter to darkness is a lantern support
  pet, not brighter caves. Hell 4–5 are near-black abyss layers.
- **Depth = desirability.** Every layer down/up raises enemy levels (+9 per depth), coin
  multipliers (up to ~17×), and rarity of content. The deepest layers are endgame.
- **Trials** (procedurally generated CoH-style door missions) launch from gates inside
  the realm layers. Their interiors are generated by code in a reserved coordinate band —
  the builder never makes mission interiors.

Art direction anchor: low-poly stylized (Synty-adjacent), saturated, readable at a
distance, mobile-friendly. The existing trial interiors (torch-lit hell, bright heaven)
are the in-house quality bar.

## 2. World architecture: ONE layout, ELEVEN skins

The world is **one layout replicated eleven times**, stacked vertically:

```
Heaven_5   Y = +10000   (empyrean apex — pure radiance)
Heaven_4   Y =  +8000
Heaven_3   Y =  +6000
Heaven_2   Y =  +4000
Heaven_1   Y =  +2000   (first ascent — gilded)
Home       Y =      0   (the base realm — natural, vibrant)
Hell_1     Y =  -2000   (first descent — ember)
Hell_2     Y =  -4000
Hell_3     Y =  -6000   (jagged, demon-infested, still visible)
Hell_4     Y =  -8000   (custom sky, no sun — dark)
Hell_5     Y = -10000   (the abyss — near-black; lantern territory)
```

**The layout is IDENTICAL on every layer.** Same footprint, same walkable paths, same
marker positions. Only the *dressing* changes: materials, props, foliage, color story,
architectural flavor. This is deliberate — players always know where things are, configs
(zones, leashes, spawners) replicate mechanically, and the builder's work is one layout
plus ten reskins instead of eleven worlds.

Per-layer skyboxes and lighting/atmosphere are **already configured in code**
(`configs/layers.lua` + the client atmosphere system). The builder designs geometry *to*
each layer's palette; the engine applies the sky and color grade.

### The layout: hub-and-spoke (new direction)

Replace the current ring with a **hub and four spokes**:

- **HUB (center):** the social/service core. Player spawn, the realm portal plaza (see
  §4.2), egg hatcher court, shops/enchanter/altar anchors, daily-reward chest, trade
  space. Compact and readable — a new player should see every affordance within one
  camera sweep.
- **FOUR SPOKES (the biomes):** Earth (grass), Ice, Lava, Desert — one element each,
  radiating from the hub. Each spoke: a crystal-farming field (the playable surface), an
  enemy presence, an egg hatcher, and a gate at its mouth (zone unlock). Spokes get
  progressively longer/wilder away from the hub.
- Spokes must be **visually distinct at a glance** (silhouette + color), because the
  element drives the combat RPS and drop branding.
- On realm layers the same spokes wear the layer's theme *tinted by their element*
  (e.g. Hell_2's Ice spoke = black ice; Heaven_1's Lava spoke = golden magma).

Scale guardrails: pets are big (huge pets up to ~6× scale, mission bosses larger) and
squads are 10+ models per player. Paths comfortably 30+ studs wide; farming fields open;
no pinch-points under ~14 studs (portal prompt range).

## 3. Hard technical rules (non-negotiable)

1. **Everything anchored, zero scripts.** The map ships as geometry + markers only. No
   Scripts, LocalScripts, ModuleScripts, no unanchored parts, no constraints doing work.
2. **StreamingEnabled is ON** (with `PauseOutsideLoadedArea` integrity). Build for it:
   modular chunks, no single mega-mesh for a whole spoke, LODs where offered.
3. **Reserved coordinate bands — leave them EMPTY:**
   - `X ≥ 24000` at any Y: procedural mission instance slots.
   - `X = 20000` band: Studio demo stamps.
4. **Y bands are exact.** Each layer folder's content lives at its band (±2000 per
   depth). We position the clones; the builder delivers layouts at origin (see §6).
5. **Folder contract:** each layer is one folder under `Workspace.Maps`, named exactly
   `Home`, `Heaven_1`…`Heaven_5`, `Hell_1`…`Hell_5`.
6. **MeshPart naming gotcha:** our import pipeline names MeshParts by the mesh DATA name,
   not the file name. Name meshes meaningfully in the DCC tool before export.
7. **No baked lighting assumptions.** Lighting/sky/fog/color-grade come from config per
   layer. Test your dressing under the layer's palette (we provide a preview place with
   the atmosphere applied — same map, all eleven grades).
8. **Group-owned assets.** Every uploaded mesh/texture must be uploaded to the game's
   group (15872767) — assets on personal accounts break for other team members.

## 4. The binding contract (what the map must carry)

Everything below is either a **CollectionService tag + attributes** (see
`docs/MAP_MARKER_REFERENCE.md` for full schemas) or a **reserved name**. Stamp helper
scripts exist for bulk tagging (`scripts/studio/stamp_authored_*.luau`).

### 4.1 Zones & spawns (per layer)

| Hook | Form | Notes |
| --- | --- | --- |
| Area volumes | `AreaZone` tag + `AreaId` | One per hub + per spoke. Ids come from `configs/areas.lua` (`Spawn`, `Grass`, `Ice`, `Lava`, `Desert` pattern). Player area resolution raycasts these. |
| Player spawn | `PlayerSpawn` tag + `AreaId` on the hub's SpawnLocation | One per layer (hub). |
| Farming surfaces | `SpawnZone` tag on the actual walkable surface mesh per spoke, `SurfaceOnly = true`, clearance attrs | This is where crystals spawn. The surface must be a real queryable mesh. |
| Zone unlock gates | `Gate` model at each spoke mouth | Sequential unlock, paid in the prior zone's coins. |

### 4.2 Travel (per layer)

| Hook | Form | Notes |
| --- | --- | --- |
| Realm portal plaza | Parts/Models named `Portal_Halo1`…`Portal_Halo5` and `Portal_Horn1`…`Portal_Horn5` | The ten-layer ladder. Prompt binds to the named part (Model → PrimaryPart). Back-to-back double-faced portals are supported (the client side-gates the prompts), but leave ~4+ studs between paired faces' anchor parts. |
| Trial gates | `MissionDoor` tag + `MissionId = "auto"` attribute on a door part | Place ONE per realm layer (Heaven_n / Hell_n hubs), none on Home. The gate's realm is derived from its `Maps.<Realm>_n` ancestry — keep gates inside their layer folder. |
| Teleport pads (optional) | `TeleportPad` tag | Hub ↔ spoke-end conveniences. |

### 4.3 Content anchors (per layer)

| Hook | Form | Notes |
| --- | --- | --- |
| Egg hatchers | Authored Models under `Maps/<Layer>/EggHatchers`, interaction part tagged `EggStand` (`AuthoredVisual = true`) | NEVER synthetic. One per spoke + hub court. Egg ids per layer come from us (`egg_sources` / `realm_area_eggs`). |
| Enemy cave anchors | Parts named `BaddieSpawner<Origin>` (e.g. `BaddieSpawnerLava`) inside realm layer folders | Realm patrol/cave bands key off these + their parent folder name. |
| Enemy leash regions | Named parts we reference from `configs/enemy_leash` (box = footprint, circle = disc) | Give each spoke's combat area a simple invisible leash part with a stable name. |
| Ascension Altar | The altar model (tagged) in the hub | Exists today in Home; each layer can restyle it. |
| Enchanter / shops / NPCs | `EnchanterStation`, `ShopAnchor`, `NPCAnchor` tags | Hub court. Heaven/hell shop reskins are a known follow-up. |
| POD podium | `PODPodium` tag | Hub. |
| Daily reward chest | Keep the existing model name/anchor (hub) | |

### 4.4 What the builder does NOT build

- Mission/trial interiors (procedural, reserved X band).
- Any UI, lighting rigs, skyboxes, or scripts.
- Crystals, eggs-in-world visuals for template mode, enemies, pets — all spawned by code
  onto the markers above.

## 5. Per-layer art briefs (the eleven skins)

Same layout every time; these are DRESSING briefs. Skyboxes for all eleven already exist
in config — build to harmonize with them.

| Layer | Story | Palette / materials | Notes |
| --- | --- | --- | --- |
| **Home** | The vibrant natural world | Saturated greens, honest biome colors per spoke | The first-90-seconds layer: most polish here. |
| Heaven_1 | First ascent — gilded meadows | Gold trim on natural forms, soft whites | Warm, inviting. |
| Heaven_2 | Cloudlands | Marble, white stone, drifting cloud props | Current maps' quality floor to beat. |
| Heaven_3 | Crystal gardens | Prismatic crystal, pale gold, glass | |
| Heaven_4 | Astral courts | White-gold architecture, light beams | Grand, vertical flourishes. |
| Heaven_5 | The Empyrean | Near-white radiance, floating geometry | Apex prestige — screenshot bait. |
| Hell_1 | First descent — scorched | Ash, ember cracks, charred versions of Home's props | Familiar-but-wrong. |
| Hell_2 | The furnace | Basalt, lava veins, chains, spikes | Matches existing hell trial dressing. |
| Hell_3 | Jagged spired wastes | Obsidian spires, red-glow accents | "Demon-infested hellscape" sky exists. |
| Hell_4 | The gloaming | Desaturated near-dark, isolated ember lights | Sun/moon disabled. Light sources scarce ON PURPOSE. |
| Hell_5 | The abyss | Near-black, silhouette-only geometry, rare red glows | Lantern-pet territory. Readability comes from emissive accents, not ambient light. |

Element tinting per spoke persists through every layer (Ice spoke stays "the ice one"
even in Hell_5 — black ice, not generic darkness).

## 6. Process & deliverables

**Phase 0 — Graybox the layout (1 file).** Hub + four spokes, blocked in primitives at
origin, all §4 markers placed. We run the binding boot + playtest flow on it. Iterate
here — moving a spoke costs nothing yet.

**Phase 1 — Dress Home.** Full art pass on the base layer only. This is the advertising
gate: when Home is good, we ship it alone (realm layers keep old dressing behind their
level gates) and start marketing while the reskins proceed.

**Phase 2 — Reskin ladder.** One layer at a time, Heaven_1 and Hell_1 first (most
players see them), apex layers last. Each reskin = the SAME layout file re-dressed;
markers must not move (we diff marker transforms in acceptance).

**Delivery format:** one `.rbxl` (or Models-folder `.rbxm`) per phase, everything under a
single `Maps/<LayerName>` folder, built at origin. We handle Y-banding, config wiring
(zones/leashes/eggs), and the atmosphere previews.

**Acceptance per delivery (we run, builder gets the report):**
1. Boot with `map_binding` validation — zero Warn/Error from WorldBindingService.
2. Hook census matches the layer manifest (every tag present, counts right).
3. Crystal spawner surface test (spawn 50, zero in props/void).
4. Walk test: spawn → each spoke → each portal → trial gate prompt appears.
5. Streaming test: warp between layers, no fall-through, no >2s integrity pause.
6. Perf: layer under budget (parts/triangles TBD with builder), 60fps on a mid phone.
